# answers.md — mixed-runtime-project

## ① 기대 탐지 결과

- 하네스 1개로 묶여야 한다. 구성 요소: `.claude/skills/report/SKILL.md`(클로드 코드 스킬, `out/data.json`을 씀) + 루트 `AGENTS.md`(코덱스 지시문, `out/data.json`을 읽어 `out/summary.md`를 만듦).
- 연결 근거는 두 가지가 겹친다(스펙 "하네스 묶기" 1번 — 둘 중 하나만 있어도 충분하지만 여기는 둘 다 있음):
  - **공유 상태** — report가 쓰는 `out/data.json`을 AGENTS.md 지시가 읽음(서로 이름은 안 불러도 데이터가 흐름).
  - **직접 참조** — `AGENTS.md`가 "Claude Code의 report 스킬이 ... 만들어 둔 파일"이라고 명시적으로 지칭함.
- 클로드 코드 아티팩트와 코덱스 아티팩트가 섞인 프로젝트이므로, 두 런타임이 각각 별개 하네스나 외톨이로 분리되어 나오면 실패다(스펙 표 "클로드+코덱스 섞인 프로젝트 → 두 런타임 아티팩트가 한 하네스로 묶임").

## ② 심은 위반

없음(스펙 지시에 따라 이 픽스처는 위반을 심지 않음). 두 파일 모두 가이드를 충족하도록 작성됨.

채점 기준: 심은 위반(여기선 없음)이 전부 잡히는 것이 필수 조건이다. 정답지에 없는 추가 지적이 나왔다고 곧바로 실패는 아니다 — 채점자가 픽스처 원본과 가이드를 대조해, 실제로 원칙에 걸리는 정당한 발견이면 통과로 인정하고, 근거 없이 억지로 지어낸 지적이면 오탐으로 실패 처리한다. (가이드는 원칙을 보는 관점이 여러 개라 픽스처를 완벽한 클린 상태로 보장하기 어렵다 — 정당한 발견까지 실패로 몰면 스킬의 올바른 동작을 벌주게 된다.)

## ③ 기대 리포트 행동

구조 다이어그램에 스킬 노드(`report`)와 문서 노드(`AGENTS.md`의 리포트 요약 작성 지시) 두 개, 그리고 그 사이를 잇는 "공유 상태" 종류의 연결선(경유: `out/data.json`)이 표시돼야 한다. 상태파일(`out/data.json`)이 노드 종류 중 "상태파일"로 별도 표시되면 다섯 질문("뭐가 있나") 요구에 더 잘 맞는다. 실행 순서상 report(1단계, 데이터 생성) → AGENTS.md 지시(2단계, 요약 생성)로 단계 번호가 매겨져야 한다.
