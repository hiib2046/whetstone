# 클로드 코드 아티팩트 탐지 카탈로그

이 문서는 `harness-inspect` 스킬이 **클로드 코드(Claude Code) 런타임**에서 하네스 부품을 "모양"으로 알아보기 위한 참고 자료다. 경로를 하드코딩해 찾지 말고, 여기 적힌 모양(파일 구조·프론트매터·참조 방식)에 맞는 후보를 Glob·Grep으로 찾는다. 항목마다 **알아보는 모양**(파일 자체를 어떻게 식별하나) · **연결 근거로 쓸 흔적**(3단계 묶기에서 다른 부품과 엮을 때 무엇을 근거로 삼나) · **주의점**(오탐·누락하기 쉬운 지점)을 담았다.

## 스킬

**알아보는 모양:** 파일명이 정확히 `SKILL.md`이고, 최상단에 YAML 프론트매터가 있으며 그 안에 `name`·`description` 키가 있다. 위치 관례는 프로젝트 `.claude/skills/<스킬이름>/SKILL.md`, 사용자 홈 `~/.claude/skills/<스킬이름>/SKILL.md`, 플러그인이면 `<플러그인루트>/skills/<스킬이름>/SKILL.md`(설치되면 `~/.claude/plugins/cache/<마켓플레이스>/<플러그인>/<버전>/skills/` 아래 캐시된다). `description`은 보통 트리거 문구("~할 때 쓴다")를 담고 있어 다른 스킬·에이전트와의 위임 관계를 읽는 데도 쓸 수 있다.

**연결 근거로 쓸 흔적:** SKILL.md 본문이 같은 폴더 안의 `scripts/`·`references/`·`agents/`·`fixtures/` 등을 상대 경로로 가리키면 그 파일들은 이 스킬의 부속이다(외톨이 후보에서 빠지고 스킬에 종속). 다른 스킬 이름이나 에이전트 이름을 본문에서 직접 부르면 그쪽과의 참조 관계.

**주의점:** 같은 폴더에 `SKILL.md`가 아닌 다른 `.md`가 있다고 전부 무시하면 안 된다 — 그 파일들이 참조 문서인지 즉석 디스패치 에이전트인지는 아래 두 항목으로 갈라 봐야 한다. 스킬이 비활성 플러그인 소속이면(플러그인이 `enabledPlugins`에서 꺼져 있으면) 실제로는 로드되지 않는다는 점도 판정에 참고할 수 있다.

## 정식 등록 에이전트

**알아보는 모양:** YAML 프론트매터가 있는 `.md` 파일로, 프론트매터에 최소 `name`·`description`이 있다. 그 외 `model`·`tools`·`color`·`skills` 같은 키가 같이 오는 경우가 흔하다. 예:

```yaml
---
name: codex-rescue
description: Proactively use when Claude Code is stuck, ...
model: sonnet
tools: Bash
skills:
  - codex-cli-runtime
---
```

위치 관례는 프로젝트 `.claude/agents/*.md`, 사용자 홈 `~/.claude/agents/*.md`, 플러그인이면 `<플러그인루트>/agents/*.md`. 이 관례 위치에 있고 프론트매터에 `name`·`description`이 채워져 있으면 클로드 코드가 이름으로 직접 호출 가능한(Agent 도구의 "Available agent types" 목록에 오르는) 정식 등록 에이전트로 본다.

**연결 근거로 쓸 흔적:** `skills:` 키가 있으면 그 스킬들과의 참조 관계가 프론트매터에 명시된 것이므로 확실한 연결이다. 그 외엔 다른 스킬·에이전트·훅 스크립트가 본문에서 이 에이전트 이름을 직접 부르는지 본다.

**주의점:** 프론트매터가 없거나 `name`·`description` 중 하나라도 비어 있으면 이 항목이 아니라 아래 "스킬 즉석 디스패치 에이전트"일 가능성을 먼저 확인한다. 관례 위치(`agents/`) 밖에 있어도 프론트매터 형태가 맞으면 후보에서 버리지 않는다 — 위치는 관례일 뿐 판정 기준이 아니다(2단계 원칙: 모양으로 찾는다).

## 스킬 즉석 디스패치 에이전트

**알아보는 모양:** 스킬 폴더 안(전형적으로 `agents/*.md`, `subagents/*.md`)의 `.md` 파일로, YAML 프론트매터가 **없다** — 첫 줄부터 바로 본문(제목·역할 설명 등)이다. 이 파일 하나만 봐서는 등록 에이전트인지 그냥 참고 문서인지 구별되지 않는다 — **반드시 그 스킬의 SKILL.md 본문을 같이 확인**해서, 본문이 이 파일을 "서브에이전트를 스폰해 이 파일을 프롬프트로 넘겨라" 식으로 가리키는지 봐야 한다. 예(SKILL.md 본문 표현):

> "spawn a grader subagent ... that reads `agents/grader.md`"
> "Read `agents/comparator.md` and `agents/analyzer.md` for the details."

**연결 근거로 쓸 흔적:** 이 항목은 정의상 이미 그 스킬과 확실히 연결돼 있다(스킬 본문의 직접 참조가 곧 식별 조건이므로 별도로 찾을 필요가 없다). 그 파일이 또 다른 스크립트·상태 파일을 언급하면 그것도 같은 하네스로 묶을 근거가 된다.

**주의점 — 오탐이 가장 잦은 지점:** 프론트매터 없는 `.md`가 스킬 폴더 안에 있고 SKILL.md가 그 경로를 언급한다고 전부 디스패치 에이전트는 아니다. **"스폰해라/서브에이전트로 실행해라/에이전트를 띄워라"처럼 실행을 지시하는 표현**과 **"읽어라/참고해라"처럼 단순 참조를 지시하는 표현**을 구별해야 한다. 후자는 스킬이 자기 컨텍스트에서 직접 읽어 쓰는 참고 문서(레퍼런스)일 뿐, 별도 세션·서브에이전트로 실행되지 않는다 — agent-guide.md 기준의 판정 대상이 아니다. 판단이 애매하면(예: "이 내용을 바탕으로 판단하라"처럼 실행 주체가 불분명) 추측으로 단정하지 말고 근거를 그대로 적어 외톨이로 남긴다.

## 훅

**알아보는 모양:** `settings.json` 또는 `settings.local.json`(위치는 프로젝트 `.claude/`, 사용자 홈 `~/.claude/`) 안의 최상위 `hooks` 키, 또는 플러그인의 `<플러그인루트>/hooks/hooks.json`(최상위가 바로 `hooks` 키인 별도 파일). 구조는 이벤트 이름(`PreToolUse`·`PostToolUse`·`SessionStart`·`SessionEnd`·`Stop`·`SubagentStop`·`Notification`·`UserPromptSubmit`·`PreCompact` 등)을 키로 하고, 값은 `{ matcher?, hooks: [{ type: "command", command: "...", timeout? }] }` 배열이다. `PreToolUse`/`PostToolUse`는 보통 `matcher`(예: `"Bash"`, `"Edit|Write"`)로 대상 도구를 좁히고, 그 외 라이프사이클 이벤트는 `matcher` 없이 전체에 걸린다. 예(사용자 홈 `settings.json`):

```json
"hooks": {
  "Stop": [
    { "hooks": [ { "type": "command", "command": "node \"C:/Users/.../discord-notify.js\" stop" } ] }
  ]
}
```

플러그인 훅은 `command`에 `${CLAUDE_PLUGIN_ROOT}` 환경변수를 써서 플러그인 설치 경로를 이식성 있게 가리킨다.

**연결 근거로 쓸 흔적:** `command`가 가리키는 스크립트 파일 자체가 이 훅의 부속이다 — 반드시 따라가 읽는다. 그 스크립트가 특정 스킬·에이전트의 산출물(상태 파일 등)을 읽거나 검사하면 그 스킬·에이전트와 확실한 연결(공유 상태 또는 직접 참조)이다.

**주의점:** `settings.local.json`은 보통 개인용·gitignore 대상이라 팀에 공유되지 않는다 — 존재하면 후보에 포함하되 "로컬 전용"이라는 성격을 판정에 참고한다. 플러그인 `hooks.json`은 그 플러그인이 `settings.json`의 `enabledPlugins`에서 켜져 있어야 실제로 동작한다 — 파일이 있어도 꺼져 있으면 죽은 훅일 수 있다. `command` 문자열이 스크립트 호출이 아니라 셸 파이프라인 전체(`bash -c '...'`)로 되어 있는 경우도 있으니, 문자열 전체를 읽고 실제로 실행되는 스크립트/커맨드를 파악한다.

## 스크립트(독립세션·코드형)

**알아보는 모양:** `.mjs`·`.py` 등 코드 파일 안에서 `claude -p`, `claude --print` 같은 CLI 호출이나, Claude Agent SDK(`@anthropic-ai/claude-agent-sdk`의 `query(...)`, Python `claude_agent_sdk`의 `ClaudeSDKClient` 등)로 **새 격리된 세션을 코드가 기동**하는 호출이 있다. 훅의 `command` 스크립트와 달리, 이 유형은 현재 대화의 도구 호출 생명주기에 동기적으로 반응하는 게 아니라 완전히 새로운 대화 세션을 통째로 새로 연다.

**연결 근거로 쓸 흔적:** 스크립트가 프롬프트를 만들 때 참조하는 파일(템플릿·설정)이나, 실행 결과를 쓰는 출력 파일이 다른 부품과 겹치면 공유 상태로 묶는다. 이 스크립트를 호출하는 쪽(훅의 `command`, 다른 스킬의 실행 지시)도 확인해 누가 이 스크립트를 기동하는지 추적한다.

**주의점:** `claude`라는 단어가 스크립트에 있다고 다 이 유형은 아니다(주석·문자열 상수일 수 있음) — 실제 프로세스 실행(`exec`/`spawn`/`subprocess` 등)에 쓰이는지 확인한다.

## 상태 파일

**알아보는 모양:** 특정 확장자나 이름 관례가 없다 — JSON·마크다운 등 어떤 파일이든, **두 개 이상의 다른 부품(스킬 본문·에이전트 프롬프트·훅 스크립트)이 그 파일 경로를 언급하며 읽거나 쓴다**는 사실 자체가 식별 조건이다. 그래서 후보 목록화 단계에서는 따로 찾지 않고, 3단계 "묶기"에서 다른 부품들의 참조를 대조하며 드러난다.

**연결 근거로 쓸 흔적:** 이 항목 자체가 곧 연결 근거다(공유 상태) — 부품 A와 B가 같은 파일 경로를 언급하면 그 파일이 A·B를 묶는 근거가 된다.

**주의점:** 파일 이름에 "state"·"progress" 등이 들어 있어도, 실제로 읽고 쓰는 부품이 하나뿐이면(다른 부품이 참조한 흔적이 없으면) 상태 파일로 묶을 근거가 없다 — 그 부품 하나의 사적 산출물로 본다.

## CLAUDE.md

**알아보는 모양:** 파일명이 정확히 `CLAUDE.md`. 프로젝트 루트, 하위 디렉토리(도메인/모듈 범위), 사용자 홈(`~/.claude/CLAUDE.md`, 전역) 어디에나 있을 수 있다. `@경로` 표기로 다른 `.md` 파일을 임포트할 수 있다 — 임포트된 파일도 같은 지시문 체인의 일부다.

**연결 근거로 쓸 흔적:** CLAUDE.md가 특정 스킬·에이전트·훅의 이름이나 사용 규칙을 직접 언급하면 그 부품과 직접 참조 관계다. 하위 디렉토리 CLAUDE.md가 그 디렉토리 안의 스킬·스크립트를 규정하면 그 범위의 하네스와 묶는다.

**주의점:** CLAUDE.md는 "그냥 문서"가 아니라 **모든 세션에 항상 로드되는 지시문 층**이라는 점에서 그 자체로 하네스 부품이다 — 스킬·에이전트·훅처럼 이 저장소 가이드(특히 `common-guide.md`·`harness-guide.md`)로 판정 대상에 넣는다. 내용이 코딩 스타일 같은 하네스와 무관한 규칙뿐이면 판정에서 "이 부분은 하네스 설계와 무관"이라고 정직하게 적고 억지로 지적을 만들지 않는다.
