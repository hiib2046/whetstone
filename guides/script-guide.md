# 좋은 하네스 동작 스크립트란

훅·스킬·에이전트·하네스의 동작 스크립트는 **운영체제에 가능한 한 종속되지 않고, 상단 주석만으로 목적·입출력·부작용·종료 결과를 알 수 있어야 한다.**

이 문서는 [common-guide.md](common-guide.md)와 스크립트가 속한 부품의 가이드([skill-guide.md](skill-guide.md)·[agent-guide.md](agent-guide.md)·[harness-guide.md](harness-guide.md))를 함께 적용한다.

## 적용 범위는 역할로 가른다

훅·스킬·에이전트 또는 하네스의 동작을 구현하는 실행 스크립트라면 파일 위치와 관계없이 적용한다. 애플리케이션 소스와 하네스 실행에 사용되지 않는 개발 보조 스크립트는 제외한다.

- **점검:** 이 파일은 훅·스킬·에이전트·하네스 실행에 사용되는가? 아니라면 애플리케이션 소스나 개발 보조 스크립트로 제외했나?

## 기본은 Node.js, Python은 전용 의존성이 있을 때만

런타임 선택을 일관되게 하기 위해 **Node.js ESM(`.mjs`)을 기본**으로 한다. 핵심 동작이 Node.js로 대체하기 곤란한 Python 라이브러리에 직접 의존할 때만 Python(`.py`)을 쓴다.

- **점검:** Python을 썼다면 핵심 동작이 Node.js로 대체하기 곤란한 Python 라이브러리를 실제로 요구하나?

## 주석은 상단과 본문으로 나눠 쓴다

모든 스크립트는 Node.js의 파일 상단 블록 주석 또는 Python의 모듈 docstring을 둔다. 상단에는 현재 동작의 목적·흐름·입출력·부작용·종료 결과와 실행 조건을 설명한다.

본문 주석은 함수나 논리 블록이 **무엇을 하고 왜 필요한지** 함께 설명한다. 자명한 코드나 이름을 번역하거나 상단 주석을 반복하지 않는다.

- **점검:** 상단 주석만으로 전체 동작과 결과를 이해할 수 있나? 본문 주석이 무엇을 하는지와 그 이유를 함께 설명하나? 코드 번역이나 상단 반복이 있나?

## 텍스트·경로·프로세스를 운영체제에 종속되지 않게 처리한다

새로 작성하는 스크립트와 스크립트가 생성하는 텍스트 파일은 BOM 없는 UTF-8과 `LF(\n)`를 기본으로 한다. 기존 파일은 원래 형식을 보존하고, 새 생성 파일에 소비자 계약이 있으면 그 계약을 따른다. 파일 입출력에는 인코딩을 명시하고(UTF-8은 Node.js `utf8`, Python `encoding="utf-8"`), 스크립트가 실행한 외부 프로그램과 텍스트를 주고받을 때도 인코딩과 오류 처리 방식을 명시한다. 바이너리에는 적용하지 않는다.

경로는 Node.js의 `node:path`, Python의 `pathlib`으로 처리한다. 스크립트와 함께 제공된 리소스는 스크립트 위치, 입력받은 상대 경로는 상단 주석에 적은 위치를 기준으로 찾는다. 외부 명령은 실행 파일과 인자 배열로 호출하고 셸을 거치지 않는다. 파이프·리다이렉션·glob은 Node.js 또는 Python API로 처리한다.

운영체제별 차이는 자동으로 분기한다. 지원하지 않는 운영체제는 상단 주석에 적고 오류로 종료한다.

- **점검:** 텍스트 입출력 형식을 명시하고 기존 형식·소비자 계약을 보존했으며 바이너리는 제외했나? 경로 기준과 외부 명령 호출이 셸·운영체제에 종속되지 않나? 운영체제 차이와 미지원 범위를 처리했나?

## 출력과 실패도 호출 계약이다

stdout은 상단 주석에 설명한 결과나 프로토콜에만 쓰고, 로그와 진단은 stderr로 분리한다. 비밀값은 출력하지 않는다. 실행 전에 확인할 수 있는 입력과 도구는 파일을 바꾸기 전에 검사한다. 스크립트 자체의 실패는 비정상 종료 코드로 알린다. 스크립트가 실행한 외부 프로그램의 실패를 별도로 처리하지 않았다면 그 종료 코드를 전달한다. 부분 상태는 정리하거나 상단 주석에 남는 상태와 재실행 방법을 적는다.

- **점검:** stdout에 결과나 프로토콜이 아닌 로그가 섞였나? 비밀값이 노출되나? 외부 프로그램의 실패를 별도로 처리하지 않았다면 그 종료 코드를 전달하나? 부분 상태가 정리되거나 설명됐나?

## 코드와 주석은 같은 변경에서 맞춘다

스크립트를 생성하거나 수정한 뒤에는 상단과 본문 주석을 코드에 대조한다. 목적·흐름·입출력·파일 부작용·성공·실패 조건·종료 코드 등이 달라졌다면 같은 변경에서 설명도 고친다. 설명을 고치지 않았다면 이 항목들에 영향이 없음을 확인한다. 낡은 주석은 제거한다.

- **점검:** 주석이 실제 인터페이스·부작용·종료 결과를 반영하나? 새 분기나 제거된 동작 때문에 틀린 주석이 생겼나? 설명을 고치지 않았다면 문서화된 동작에 영향이 없음을 확인했나?

## 변경한 스크립트를 검증한다

완료 보고가 실제 동작을 근거로 하도록 구문 검사와 대표 실행을 수행한다. 실행하지 못한 검사는 완료 보고에 범위와 이유를 적는다.

- **점검:** 구문 검사와 대표 실행을 했나? 실행하지 못한 검사의 범위와 이유를 보고했나?
