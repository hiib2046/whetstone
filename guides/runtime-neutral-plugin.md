# 런타임 중립 플러그인이란

런타임 중립 플러그인은 Claude Code·Codex·Hermes처럼 서로 다른 AI 런타임에서 쓰되, 핵심 동작을 런타임별로 복제하지 않는 플러그인이다. **공통 핵심은 하나만 두고, 런타임 차이는 얇은 어댑터에서 연결한다.** 그래야 기능을 고칠 때 공통 핵심 한 곳만 수정하고도 모든 지원 런타임에 같은 변경을 반영할 수 있다.

이 가이드는 여러 런타임에 설치하는 **플러그인**의 구조를 다룬다. 특정 작업 저장소 안에서만 쓰는 프로젝트 전용 하네스의 설치 위치와 배포 방식은 범위 밖이다. 또한 각 런타임의 설치 명령이나 매니페스트 문법을 정하지 않는다. 그런 파일·필드 형식의 차이는 해당 런타임의 어댑터와 등록 파일이 담당한다.

## 공통 핵심과 어댑터의 경계를 지킨다

여러 런타임에서 의미와 동작이 같은 스킬 본문·에이전트 본문·참고 문서·훅 로직은 공통 핵심에 한 번만 둔다. 어댑터에는 실제 도구 식별자, 훅 이벤트명, 매니페스트 필드, 런타임 입출력 형식처럼 **해당 런타임에 연결할 때만 필요한 차이**를 둔다. 어댑터가 공통 로직을 다시 구현하면 같은 기능의 수정 지점이 여러 곳으로 늘어난다.

공통 문서에는 별도의 추상 용어 체계나 런타임별 용어 해석표를 만들지 않는다. 모델이 의미만으로 자기 런타임의 기능을 선택할 수 있으면 특정 도구명 대신 분명한 자연어로 지시한다. 정확한 식별자·필드·프로토콜이 있어야 실행되는 부분만 어댑터로 보낸다. 모든 런타임에서 실제 상대 경로가 같은 파일은 그대로 가리키며, 경로까지 추상화하지 않는다.

- **점검:** 공통 핵심은 한 곳에만 있는가? 어댑터는 런타임 연결과 입출력 변환만 담당하고 공통 동작을 다시 구현하지 않는가? 공통 문서는 특정 런타임의 도구명·환경 변수명·훅 이벤트명 없이 이해되는가? 자연어로 충분한 지시에는 별도 추상 용어나 해석표를 만들지 않았는가? 모든 런타임에서 같은 실제 경로는 그대로 사용하는가?

## 필요한 부품만 만든다

아래 구조는 모든 폴더를 반드시 만들라는 템플릿이 아니다. 각 부품이 필요할 때 **어디에 둘지**를 정한 것이다. 플러그인이 실제로 사용하는 부품만 만든다. 예를 들어 훅이 없으면 `scripts/hooks/`와 그 테스트도 만들지 않는다.

```text
<plugin>/
├─ skills/
│  └─ <skill>/
│     ├─ SKILL.md
│     └─ agents/                 # 아래 파일 중 필요한 것만 둔다
│        ├─ <agent>.md           # 이 스킬이 디스패치할 에이전트 본문
│        └─ openai.yaml          # Codex만 읽는 보조 파일의 예
├─ scripts/
│  └─ hooks/                     # 훅이 있을 때
│     ├─ <capability>.mjs
│     ├─ adapters/
│     │  └─ <runtime>/
│     │     ├─ <capability>.mjs
│     │     └─ initialize-hook-support.mjs
│     └─ fallbacks/
│        └─ <capability>.md
├─ <runtime registration files> # 런타임별 매니페스트·훅 등록 파일
└─ tests/
   ├─ common/
   └─ runtimes/
      ├─ adapters/
      └─ smoke/
```

- **점검:** 실제로 사용하는 부품의 폴더와 파일만 만들었는가? 공통 파일은 한 곳에만 두었는가? 런타임 전용 매니페스트와 훅 등록 파일은 공통 핵심과 구분되는 위치에 있는가?

## 스킬은 하나의 공통 폴더를 함께 쓴다

플러그인 루트의 `skills/`를 모든 런타임이 함께 사용한다. 각 `SKILL.md` 프런트매터에는 공통 필드인 `name`과 `description`만 두고, 본문 전체를 런타임 중립적인 지시로 작성한다. 여러 런타임이 같은 스킬 본문을 읽으므로 본문을 런타임별로 복사하지 않는다.

특정 런타임만 읽는 보조 파일은 같은 스킬 폴더에 함께 둘 수 있다. 예를 들어 Codex용 `agents/openai.yaml`은 다른 런타임이 자신의 설정 파일로 해석하지 않으므로 공통 `SKILL.md`를 갈라야 할 이유가 아니다. 다만 한 런타임의 전용 필드를 모든 런타임이 파싱하는 `SKILL.md` 프런트매터에 섞지는 않는다.

- **점검:** 모든 런타임이 하나의 `skills/`와 `SKILL.md` 본문을 함께 사용하는가? `SKILL.md` 프런트매터에는 `name`과 `description`만 있는가? 런타임 전용 보조 파일은 공통 본문을 나누지 않고 같은 스킬 폴더에 함께 두었는가? 스킬 본문은 특정 런타임의 도구명 없이 이해되는가?

## 서브에이전트는 스킬 디스패치로 사용한다

런타임 중립 플러그인의 스킬 전용 서브에이전트는 런타임에 정식 등록하지 않는다. 에이전트 본문을 `skills/<skill>/agents/<agent>.md`에 두고, 스킬이 필요한 시점에 그 본문을 현재 런타임의 범용 서브에이전트에 전달한다. 이렇게 하면 런타임마다 다른 에이전트 등록 형식을 공통 본문에서 제거할 수 있다.

에이전트 파일에는 자동 발견용 `name`·`description` 프런트매터나 런타임 전용 설정은 넣지 않는다. 호출할 때마다 달라지는 내용은 플레이스홀더로 표시하고 호출하는 스킬이 채운다. 크거나 가변적인 입력은 프롬프트에 붙이지 않고 파일 경로로 전달한다.

범용 서브에이전트에 프롬프트를 전달하는 방식으로는 전용 시스템 프롬프트, 모델 지정, 도구 제한을 보장할 수 없다. 이 기능들이 반드시 필요한 에이전트에는 이 공통 디스패치 구조를 적용하지 않는다. 이식성을 유지하려고 필수 기능을 조용히 제거하지 말고, 런타임 중립 구조로 지원할 수 없는 요구임을 사용자에게 알린다.

- **점검:** 스킬 전용 에이전트는 정식 등록하지 않고 스킬이 범용 서브에이전트로 호출하는가? 에이전트 본문은 `skills/<skill>/agents/`에 한 번만 존재하는가? 공통 본문에는 자동 발동용 프런트매터와 런타임 전용 설정이 없는가? 호출마다 달라지는 값은 플레이스홀더로 받는가? 큰 입력은 내용을 프롬프트에 통째로 붙이지 않고 파일 경로로 전달하는가? 전용 시스템 프롬프트·모델·도구 제한이 필요하면 지원 범위 밖임을 사용자에게 알리는가?

## 훅은 런타임마다 지원 수준을 판정한다

같은 목적의 훅이라도 런타임마다 제공하는 실행 시점(세션 시작, 도구 호출 전후 등)과 강제력이 다르다. 따라서 **훅과 런타임의 조합마다** 다음 세 수준 중 하나를 정한다.

- **정상 지원** — 동등한 네이티브 훅으로 공통 동작을 강제한다.
- **제한 지원** — 동등한 훅은 없지만 유용한 우회가 가능하여 지시문으로 대체한다. 네이티브 훅과 같은 강제력은 보장하지 않는다.
- **미지원** — 동등한 훅도 유용한 우회도 없어 동작을 연결하지 않는다.

지원 수준을 플러그인 전체에 한 번 정하지 않는 이유는 같은 런타임에서도 어떤 훅은 정상 지원하고 다른 훅은 미지원일 수 있기 때문이다. 제한 지원과 미지원을 정상 지원처럼 표시하면 사용자가 보장 수준을 잘못 이해하므로 실패다.

- **점검:** 훅과 런타임의 조합마다 지원 수준을 따로 정했는가? 같은 실행 시점과 강제력을 제공하는 네이티브 훅이 있을 때만 정상 지원으로 판정했는가? 지시문 우회는 네이티브 훅보다 보장 수준이 낮다고 표시했는가? 유용한 우회가 없으면 미지원으로 판정했는가?

## 정상 지원 훅은 공통 로직과 변환기를 분리한다

훅의 실제 동작은 `scripts/hooks/<capability>.mjs`에 둔다. 파일명은 런타임별 이벤트명이 아니라 `validate-output`, `protect-write`처럼 수행하는 기능을 나타낸다. 공통 스크립트는 특정 런타임의 입력이나 출력 형식을 직접 알지 않는다.

공통 입출력 계약은 모든 훅에 하나의 범용 스키마를 강제하지 않고, **각 훅을 설계할 때 그 훅에 필요한 최소 형식으로 한 번 정의한다.** 이후 그 훅을 지원하는 모든 런타임이 같은 계약을 지킨다. 계약은 공통 스크립트 상단 주석에 입력·출력·실패 조건과 함께 명시한다.

런타임별 변환기는 `scripts/hooks/adapters/<runtime>/<capability>.mjs`에 둔다. 실제 런타임의 훅 등록은 공통 스크립트가 아니라 이 변환기를 가리킨다. 변환기는 런타임 입력을 공통 계약으로 바꾸어 공통 스크립트를 호출하고, 공통 결과를 런타임의 출력 형식으로 바꿔 반환한다. 정상 지원하는 런타임에는 훅마다 대응하는 변환기가 하나씩 있어야 한다.

- **점검:** 공통 훅 스크립트는 런타임 이벤트명과 전용 필드를 직접 처리하지 않는가? 공통 입출력 계약은 훅마다 필요한 최소 필드로 한 번 정의되어 있는가? 같은 훅의 계약을 모든 런타임 어댑터가 함께 지키는가? 실제 훅은 공통 스크립트가 아니라 해당 런타임의 변환기에 연결되는가? 정상 지원하는 모든 훅과 런타임 조합에 대응 변환기가 있는가?

## 제한 지원은 공통 우회 지시문을 주입한다

제한 지원 훅의 공통 우회 지시문은 `scripts/hooks/fallbacks/<capability>.md`에 훅마다 하나씩 둔다. 이 파일은 자동 발견되는 스킬이나 에이전트가 아니므로 프런트매터를 넣지 않는다. 런타임별 값이 필요한 자리만 플레이스홀더로 두고, 지시의 의미는 모든 런타임에서 같게 유지한다.

제한 지원이나 미지원 기능이 있는 런타임에는 `scripts/hooks/adapters/<runtime>/initialize-hook-support.mjs`를 하나 둔다. 이 초기화 스크립트는 해당 런타임에서 제한 지원하는 지시문만 읽고, 플레이스홀더를 채워 하나의 모델용 지시문으로 합친다. 미지원 항목은 수행할 수 없으므로 모델 지시문에 넣지 않는다.

런타임 어댑터는 세션 시작·재개, 컨텍스트 초기화·압축처럼 모델이 이전 지시를 잃을 수 있는 시점 뒤에 초기화 스크립트의 결과를 다시 넣는다. 세션 시작·재개와 컨텍스트 초기화 뒤에는 모델용 지시문과 사용자 안내 메시지를 제공한다. 컨텍스트 압축 뒤에는 잊힌 지시문만 다시 제공하고 사용자 안내는 반복하지 않는다. 실제 런타임이 이 실행 시점 중 일부만 제공한다면 가능한 시점만 연결하고 그 한계를 사용자에게 알린다.

- **점검:** 각 훅의 우회 지시문은 `fallbacks/`에 하나만 존재하는가? 우회용 Markdown에는 프런트매터가 없는가? 초기화 스크립트는 제한 지원 지시문만 모델에게 제공하고 미지원 항목은 제외하는가? 각 런타임은 여러 우회 지시문을 초기화 스크립트 하나에서 합치는가? 세션 시작·재개와 컨텍스트 초기화 뒤에는 사용자 안내를 한 번 제공하고, 컨텍스트 압축 뒤에는 지시문만 다시 제공하는가? 실제 런타임이 제공하는 실행 시점에만 연결했는가?

## 제한과 미지원은 사용자에게 동작 시점에 알린다

사용자는 README를 읽지 않을 수 있으므로 제한 지원과 미지원은 가능한 한 플러그인이 동작할 때 안내한다. 안내 메시지에는 내부 훅 이름 대신 사용자가 체감하는 실제 기능명을 쓴다.

제한 지원 안내에는 **동등한 네이티브 훅이 없어 지시문으로 대체했다는 이유**와 **네이티브 훅처럼 강제되지 않는다는 제한**을 함께 적는다. 미지원 안내에는 **동등한 훅과 유용한 우회가 없어 해당 기능이 작동하지 않는다는 결과**를 적는다. 여러 항목은 런타임 초기화 때 하나의 메시지로 합친다. 런타임에 자동 안내 수단 자체가 없을 때만 사용자 문서로 대신한다.

- **점검:** 제한 지원과 미지원 기능을 플러그인이 동작할 때 사용자에게 안내하는가? 안내에는 내부 훅 이름이 아니라 영향을 받는 실제 기능명이 있는가? 제한 지원에는 지시문 대체라는 이유와 강제되지 않는다는 한계가 있는가? 미지원에는 해당 기능이 작동하지 않는다는 결과가 있는가? 여러 항목은 하나의 메시지로 합치는가? 자동 안내가 불가능할 때만 사용자 문서로 대신하는가?

## 공통 동작과 런타임 연결을 나누어 검증한다

공통 핵심은 런타임과 독립적으로 한 번 검증하고, 각 런타임에서는 공통 로직을 반복 시험하기보다 어댑터와 실제 등록이 올바르게 이어지는지 검증한다. 런타임별 내부 실행 방식은 달라도 되지만 외부에서 관찰되는 결과와 선언한 지원 수준은 일치해야 한다.

```text
tests/
├─ common/
│  ├─ structure.test.mjs
│  ├─ prompts.test.mjs
│  └─ hooks/
│     └─ <capability>.test.mjs
└─ runtimes/
   ├─ adapters/
   │  └─ <runtime>/
   │     └─ <capability>.test.mjs
   └─ smoke/
      ├─ scenarios.md
      └─ <runtime>.md
```

`tests/common/`은 공통 파일 구조와 프런트매터, 런타임 전용 표현이 들어갔는지, 플레이스홀더 치환, 공통 훅 계약을 검사한다. `tests/runtimes/adapters/<runtime>/`은 실제 런타임을 실행하지 않고 대표 입력으로 변환기의 입력 변환과 출력 변환을 검사한다.

`tests/runtimes/smoke/scenarios.md`는 런타임과 무관한 실제 실행 시나리오와 통과 조건을 한 번만 기록하는 공통 원본이다. 런타임별 `<runtime>.md`에는 시나리오를 복사하지 않고 실행 방법과 예상 지원 수준만 적는다. 자동화가 안정적인 런타임에만 실행 스크립트를 추가한다.

같은 발견·호출 구조를 공유하는 스킬은 대표 하나를 실제 실행하고, 나머지는 실행 없이 파일 구조와 지시문 형식을 검사한다. 정상 지원 훅은 모든 런타임 어댑터의 실제 이벤트 연결을 확인한다. 제한 지원은 초기화 때 지시문과 사용자 안내가 제공되고 컨텍스트 압축 후에는 지시문만 복원되는지 확인한다. 미지원은 사용자 안내가 제공되는지 확인한다.

- **점검:** 공통 핵심은 `tests/common/`에서 한 번 검증하는가? 각 런타임 테스트는 어댑터와 실제 등록의 연결에 집중하는가? 어댑터의 입력 변환과 출력 변환을 모두 검사하는가? 실제 실행 시나리오는 `scenarios.md` 한 곳에만 있는가? 제한·미지원 선언과 실제 지시문·사용자 안내가 일치하는지 검사하는가?

## 지원 표시는 관찰 가능한 검증 뒤에만 한다

다음 항목이 모두 통과한 런타임만 플러그인이 지원한다고 표시한다.

- 플러그인이 해당 런타임에 설치되고 발견된다.
- 대표 스킬이 발견되어 실행된다.
- 스킬 디스패치 에이전트가 있다면 대표 호출이 완료된다.
- 정상 지원으로 선언한 모든 훅 어댑터가 실제 이벤트에 연결된다.
- 제한 지원 지시문과 제한·미지원 사용자 안내가 선언대로 제공된다.

지원 표시는 “파일이 존재한다”가 아니라 사용자가 관찰할 수 있는 실행 결과를 근거로 한다. 일부 항목을 검증하지 못했다면 전체 지원으로 표시하지 않고 검증하지 못한 범위를 밝힌다.

- **점검:** 설치와 플러그인 발견을 확인했는가? 대표 스킬과 해당하면 에이전트 디스패치를 실제로 실행했는가? 정상 지원으로 선언한 모든 훅의 실제 이벤트 연결을 확인했는가? 제한·미지원 안내가 선언대로 제공되는지 확인했는가? 검증하지 못한 범위가 있다면 지원 표시에서 제외하거나 그 범위를 밝혔는가?
